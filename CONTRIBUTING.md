# Contributing to ViVi Music

## Governance & Standards
We enforce "Iron-Clad" quality standards to ensure the project remains maintainable and scalable.

### 1. Code Style
-   **Kotlin**: We use **Ktlint** to enforce the official [Android Kotlin Style Guide](https://developer.android.com/kotlin/style-guide).
-   **Formatting**: You MUST run `./gradlew ktlintFormat` before submitting any PR. CI will fail if lint check fails.
-   **Naming**:
    -   Classes/Objects: `PascalCase`
    -   Functions/Properties: `camelCase`
    -   Constants: `UPPER_SNAKE_CASE`

### 2. Architecture
-   **Single Activity**: `MainActivity` is the only UI entry point.
-   **SFC (Single File Components)**: Colocate your UI, private previews, and helper composables in one file.
-   **Barrels**: Use visibility modifiers (`private`/`internal`) to hide implementation details. Only expose what is necessary.

### 3. Testing (TDD)
-   **Coverage Goal**: 80% line coverage for business logic.
-   **Unit Tests**: Use **MockK** and **Turbine** for ViewModels and Repositories.
-   **UI Tests**: Use Compose Test Rule for atomic UI verification.

### 4. Pull Request Process
1.  **Format**: Run `./gradlew ktlintFormat`.
2.  **Test**: Run `./gradlew testDebugUnitTest`.
3.  **Title**: Use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/).
    -   `feat: add new player control`
    -   `fix: resolve crash on startup`
    -   `refactor: migrate NewsDetailScreen to SFC`

### 5. Resources
-   **Strings**: All strings must go into `strings.xml`. Do NOT create separate string files.
-   **Icons**: Use Vector Assets.
